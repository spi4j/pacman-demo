Bienvenue dans votre nouveau projet demo-dsfr-rest

Ce fichier vous permet de rapidement prendre en main le démarrage de l'application :

📁 Projets inclus

Quatre projets sont créés :

  • demo-dsfr-rest : le projet qui encapsule les trois autres projets, son seul rôle est de contenir le fichier pom.xml parent.
  • demo-dsfr-rest-domain : le projet métier (architecture hexagonale), il ne contient aucune référence technique et ne doit comporter que les services métiers.
  • demo-dsfr-rest-model : le projet de modélisation. Aucune classe Java ne doit être présente dans ce projet. Il n'est, par ailleurs, pas embarqué lors du déploiement de l'application.
  • demo-dsfr-rest-server : le projet d'infrastructure qui contient le framework Spring Boot et toutes les couches techniques pour manipuler la couche métier demo-dsfr-rest-domain.

📂 Emplacement des fichiers de configuration et classe de démarrage

  • Configuration des options pour la génération : demo-dsfr-rest-model/pacman-properties/project.properties Utiliser ce fichier de configuration pour modifier les différentes options utilisées pour la génération de code. Ce fichier a initialement été créé suite aux principaux choix dans le formulaire de création mais toute modification sera automatiquement prise en compte lors de la prochaine génération. Attention : lors du développement et avec la configuration proposée par défaut, l'initialisation de la base de données est effectuée directement par SpringData et non à l'aide de scripts générés.
  • Configuration du nommage des classes et interfaces pour la génération : demo-dsfr-rest-model/pacman-properties/nommage.properties Utiliser ce fichier de configuration pour modifier les différentes options de nommage utilisées pour la génération de code. Si par exemple vous désirez que tous la paramètres des méthodes soient préfixés par 'p_', il suffit de modifier la propriété suivante dans le fichier de configuration parameterMethodClassicIn=[CamelCase.LowerFirst/] en parameterMethodClassicIn=p_[CamelCase.LowerFirst/]. Toute modification sera automatiquement prise en compte lors de la prochaine génération.
  • Configuration principale : demo-dsfr-rest-server/src/main/resources/application.properties Le fichier de configuration de SpringBoot, avec l'api de santé, les metrics, les datasources, etc...
  • Configuration pour les tests d'intégration : demo-dsfr-rest-server/src/test/resources/test.properties Le fichier de configuration pour les tests de SpringBoot avec les informations pour la configuration de la datasource de test. Attention : pour les tests, l'initialisation de la base de données est effectuée à l'aide des scripts SQL générés.
  • Répertoire de stockage des scripts SQL : demo-dsfr-rest-server/src/main/resources/sql Dans ce répertoire sont stockés tous les scripts SQL de création de la base de données pour la production, il existe un script par type de base de données sélectionnée et présente dans le fichier de configuration project.properties.
  • Classe principale (à exécuter en Java Application) : demo-dsfr-rest-server/src/main/java/fr.demo_dsfr_rest/Demo_dsfr_restBootstrap La classe de démarrage de SpringBoot.

📁 Principaux packages générés

  • demo-dsfr-rest-server
    • fr.demo_dsfr_rest.app : classes de haut niveau pour les services rest.
    • fr.demo_dsfr_rest.app.adapters.controllers : contrôleur par défaut pour la customisation de l'api de santé.
    • fr.demo_dsfr_rest.app.exceptions : exceptions de base pour les services rest.
    • fr.demo_dsfr_rest.infra : classes de haut niveau pour les services de persistance.
  • demo-dsfr-rest-domain
    • fr.demo_dsfr_rest.domain : annotations métier.
    • fr.demo_dsfr_rest.domain.a.exceptions : exceptions de base pour la couche métier.
    • fr.demo_dsfr_rest.domain.services.requirements : classes de haut niveau pour les règles de gestion.

🚀 Pour tester rapidement le squelette de l'application

  1. Générer la couche SOA :

    • Clic droit sur le fichier de modélisation demo-dsfr-rest-model
    • Choisir : /Générateurs Cali → Génération de la couche de service

  2. Lancer l'application :

    • Clic droit sur la classe principale dans demo-dsfr-rest-server/src/main/java/fr.demo_dsfr_rest/Demo_dsfr_restBootstrap
    • Choisir : /Run As → Java Application

  3. Par défaut, les URIs suivantes doivent répondre :

    • http://localhost:8080/actuator/health
    • http://localhost:8080/actuator/info
    • http://localhost/swagger-ui
    • http://localhost/api-docs

🚀 Pour commencer rapidement

  1. Modéliser des entités au niveau du projet de modélisation (demo-dsfr-rest.entity)

  2. Générer les scripts SQL à l'aide du générateur SQL.

  3. Générer les classes à l'aide du générateur de la couche de persistance.

  4. Aller dans le projet demo-dsfr-rest-server et faire tourner les tests sur la datasource.

  5. Modéliser la couche SOA au niveau du projet de modélisation (demo-dsfr-rest.soa).

  6. Compléter la couche SOA en positionnant le code nécessaire au niveau des balises dédiées au code utilisateur.

  7. Aller dans le projet demo-dsfr-rest-server, compléter et faire tourner les tests d'intégration.

  8. Lancer la classe de boostrap pour l'application et tester les réponses du serveur.
